# PRD — Table for Three AI
**Product Requirements Document**
Версия 1.0 · Апрель 2026

---

## 1. Обзор продукта

**Table for Three AI** — это платформа для проведения структурированных экспериментов с многомодельными AI-разговорами. Ключевая идея: несколько языковых моделей (или несколько инстанций одной модели) участвуют в одном разговоре одновременно, каждая — в своей роли, со своим характером и системным промптом.

Флагманский эксперимент — **AI Mafia**: ролевая игра, в которой AI-участники с ролями Детектива, Невиновного и Мафии ведут диалог, стараясь оставаться неотличимыми от человека.

---

## 2. Проблема

Современные AI-чаты построены по схеме «один пользователь — одна модель». Это ограничивает:

- **Исследователей**: нельзя наблюдать, как разные модели реагируют на один и тот же контекст
- **Разработчиков промптов**: трудно сравнивать поведение одной модели с разными системными инструкциями
- **Дизайнеров агентных систем**: нет удобной среды для отладки multi-agent взаимодействий
- **Игровые/нарративные эксперименты**: отсутствие платформы для ролевых AI-сценариев с несколькими участниками

## 3. Целевая аудитория

| Аудитория | Сценарий использования |
|-----------|----------------------|
| AI-исследователи | Сравнение поведения разных моделей в одном контексте |
| Prompt engineers | Тестирование системных промптов в динамическом диалоге |
| Разработчики multi-agent систем | Прототипирование и отладка агентных взаимодействий |

| Энтузиасты AI | Эксперименты с игрой «Мафия» и другими форматами |

## 4. Цели продукта

### Главная цель
Предоставить гибкую платформу для запуска и наблюдения за разговорами между несколькими AI-агентами с настраиваемыми ролями, режимами взаимодействия и правилами поведения.

### Измеримые цели (v1.0)
- Пользователь может запустить беседу с 2–3 AI-участниками менее чем за 30 секунд
- Поддержка параллельного и последовательного режимов без изменения кода
- Ответы всех участников доставляются клиенту через WebSocket в реальном времени
- Конфигурация ролей и правил — через файлы, без перезапуска сервера

### 5.2 Режимы взаимодействия

**Параллельный режим** (`mode: parallel`)
- Все активные участники получают одно сообщение одновременно
- Ответы собираются параллельно через `asyncio.gather`
- Клиент получает ответы всех участников в одном событии
- Применение: сравнение моделей, голосование, брейншторм

**Последовательный режим** (`mode: sequential`)
- Участники отвечают поочерёдно в заданном порядке
- Каждый следующий участник видит ответы предыдущих в контексте
- Применение: ролевые игры, дебаты, цепочки рассуждений

| ID | Требование | Приоритет |
|----|-----------|-----------|
| F-07 | Параллельный режим: одновременный запрос всех участников | Высокий |
| F-08 | Последовательный режим: накопление контекста между ходами | Высокий |
| F-09 | Стриминг ответов: отправка фрагментов клиенту по мере генерации | Средний |
| F-10 | Индикация текущего говорящего в последовательном режиме | Средний |

## 6. Нефункциональные требования

### Производительность
- Время первого ответа в параллельном режиме: не более 8 секунд (таймаут буфера)
- Поддержка не менее 10 одновременных бесед без деградации

### Надёжность
- Корректная обработка разрыва WebSocket-соединения без утечек памяти
- Таймаут ожидания ответа AI: 8 секунд (конфигурируется в `workflow.json`)
- Логирование всех ошибок взаимодействия с AI API

### Безопасность
- API-ключи только через переменные среды (не в коде и не в конфиг-файлах)
- Фильтрация вредоносного контента через слой модерации
- Маскировка персональных данных в ответах

### Расширяемость
- Добавление новой AI-модели — только через `config/models.json`
- Добавление новой роли — только через `config/mafia_prompt.md` и запись участника
- Новые правила фильтрации — только через `config/workflow.json`

## 7. Пользовательские сценарии (User Flows)

### Сценарий A: Запуск эксперимента AI Mafia

```
1. Пользователь открывает веб-интерфейс
2. Система создаёт беседу с тремя участниками:
   - Детектив (Gemini, промпт из mafia_prompt.md с ролью Detective)
   - Невиновный (Gemini, промпт с ролью Innocent)
   - Мафия (Gemini, промпт с ролью Mafia)
3. Пользователь вводит первое сообщение-ситуацию
4. В последовательном режиме: Детектив задаёт вопрос → Невиновный отвечает → Мафия реагирует
5. Пользователь наблюдает развитие диалога и пытается угадать, кто из AI — Мафия
6. Раунды продолжаются до явного завершения беседы
```

### Сценарий B: Сравнение моделей в параллельном режиме

```
1. Исследователь создаёт беседу с 3 участниками, каждый — разный вариант системного промпта
2. Режим: parallel
3. Исследователь отправляет один вопрос
4. Все три участника отвечают одновременно
5. Ответы отображаются рядом — удобно для сравнения стиля и содержания
```

### Сценарий C: Агентная цепочка рассуждений

```
1. Разработчик создаёт беседу с участниками: Аналитик → Критик → Синтезатор
2. Режим: sequential
3. Каждый участник развивает мысль предыдущего
4. Итоговый ответ Синтезатора содержит объединённую позицию всей цепочки
```

## 8. Текущее состояние vs Целевое состояние

| Область | Текущее состояние | Целевое состояние (v1.0) |
|---------|------------------|--------------------------|
| Сервер | Запускается, отдаёт статику | + подключён router, работают все API-эндпоинты |
| API | `api/routes.py` написан, но не подключён к `main.py` | Роутер подключён через `include_router` |
| WebSocket | Echo-заглушка в `main.py` | Реальный WS с обработкой conversation через `ConversationManager` |
| Стриминг | Фрагменты накапливаются в буфере, не пушатся клиенту | Фрагменты ответа отправляются в WS по мере генерации |
| Фронтенд | Подключён к echo, отправляет plain text | Подключён к `/ws/{id}`, отправляет JSON, показывает ответы по участникам |
| Конфиги | Существуют, не загружаются | Загружаются при старте через `config/loader.py` |
| Модерация | Модели описаны в `models.py`, логика не реализована | Слой модерации применяется к входящим и исходящим сообщениям |
| Identity protection | Правила описаны в `workflow.json`, не применяются | Фильтр применяется к каждому ответу AI |
| Персистентность | In-memory (потери при рестарте) | Redis или SQLite для хранения состояния бесед |
| Тесты | Зависимости (`pytest`) установлены, тестов нет | Покрытие ключевых сервисов и API-эндпоинтов |

## 9. Технические ограничения и зависимости

### Внешние зависимости
- **Google Generative AI API** (`GOOGLE_API_KEY`) — обязательна для работы AI-функциональности
- **OpenAI Moderation API** (`OPENAI_MODERATION_KEY`) — опциональна, используется в `config/moderation.json`; при отсутствии модерация должна корректно деградировать

### Технический долг (критический, требует решения до v1.0)

1. **`main.py`**: добавить `app.include_router(router)` — без этого платформа не работает
2. **`api/routes.py`**: `AIService()` инициализируется при импорте — перенести в FastAPI lifespan или `Depends`
3. **`services/conversation_manager.py`**: утечка памяти завершённых бесед — добавить очистку
4. **`utils/buffer.py`**: стриминговые чанки не доходят до клиента — реализовать push через WebSocket
5. **`static/index.html`**: не интегрирован с conversation API — переписать JS-логику
